## 📌 Overview
Linked-Local Multicast Name Resolution; LLMNR (UDP 5355) and NetBIOS Name Service (NBT-NS, UDP 137) are fallback prtocols used in Windows networks when DNS resolution fails. 

Because these protocols do not authenticate responses, an attacker on the local network can impersonate a requested host and force a victim machine to authenticate to the attacker. This may expose NTLM challenge/response hashes or enable attacks such as SMB relay.

This lab simulates an **LLMNR poisoning attack** using Responder from a Kali attacker machine against two Windows clients and a Domain Controller.

Security monitoring and analysis were performed using:
- **Security Onion (Suricata IDS)**
- **Splunk (Windows Event Logs & Sysmon)**
- **Wireshark (packet analysis)**

The objective of the lab is to demonstrate:
- How LLMNR poisoning works
- What telemetry the attack generates
- How analysts can investigate the activity
- How detection rules can be engineered for SOC environments

## ⚔️ Attack Simulation

The attack was performed using Responder from a Kali Linux machine.

### Steps

1. Victim attempts to resolve a hostname.
2. DNS resolution fails.
3. Victim sends an LLMNR broadcast request.
4. Attacker responds pretending to be the requested host.
5. Victim attempts NTLM authentication to the attacker.
6. NTLM challenge/response hashes are captured.

Hashes were cracked using Hashcat.

## 🚨 Detection Engineering

Detection rules were developed to identify LLMNR poisoning activity in enterprise environments.

These include:

### Suricata Rules
Detection of suspicious LLMNR traffic and subsequent SMB authentication traffic.

### Sigma Rules
Correlation between LLMNR queries and SMB authentication activity.

### Splunk Detection Queries
Detection of authentication attempts by processes using explicit credentials.

All detection logic is available in the **detection-engineering/** directory.

---

## 🛡️ Enterprise Mitigations

Recommended mitigations include:

- Disable LLMNR via Group Policy  
  `Computer Configuration → Administrative Templates → Network → DNS Client → Turn off multicast name resolution`

- Disable NetBIOS where possible

- Enforce SMB signing

- Prefer Kerberos authentication over NTLM

- Monitor abnormal NTLM authentication attempts in SIEM platforms

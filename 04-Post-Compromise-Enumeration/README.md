## 📌 Overview

Post-compromise enumeration is a critical phase of the Active Directory attack lifecycle where an attacker, having already obtained valid credentials or remote access, begins gathering information about the compromised host and surrounding domain environment.

Attackers use this phase to:
- Identify users and systems
- Discover administrative privileges
- Enumerate network services
- Map Active Directory relationships
- Prepare for lateral movement or privilege escalation

This lab demonstrates post-compromise enumeration techniques using:
- Native Windows reconnaissance commands
- Windows Remote Management (WinRM)
- PowerView
- SharpHound/BloodHound

The attack begins with the use of compromised domain credentials obtained during previous attacks in the lab series. The attacker establishes remote access to a Windows workstation and performs host and network reconnaissance using both native operating system utilities and PowerShell-based tooling.

The attack was simulated in an Active Directory environment and monitored with:
- **Security Onion (Suricata IDS)**
- **Splunk (Windows Event Logs & Sysmon)**
- **Wireshark (packet analysis)**

The objective of the lab is to demonstrate:
- How post-compromise enumeration is performed
- The telemetry generated during host and network reconnaissance
- How analysts investigate remote PowerShell activity
- Detection opportunities for SOC environments
- Visibility limitations in endpoint and network telemetry

## ⚔️ Attack Simulation

The attack was performed from a Kali Linux attacker machine using compromised domain credentials.

### Steps

1. Attacker identifies remote administration services exposed on internal systems using Nmap.
2. Attacker attempts remote access through:
   - Remote Desktop Protocol (RDP)
   - Windows Remote Management (WinRM)
3. Successful NTLM-authenticated network logon is established.
4. WinRM launches a remote PowerShell session through `wsmprovhost.exe`.
5. Attacker executes low-noise reconnaissance commands including:
   - `whoami`
   - `hostname`
   - `systeminfo`
   - `ipconfig`
   - `net.exe`
6. Attacker attempts to deploy:
   - PowerView via the file `PowerView.ps1`
   - SharpHound/BloodHound via the ingestor file `SharpHound.ps1`
7. Enumeration activity generates endpoint and network telemetry across the environment.

## 📡 Telemetry Artifacts

The attack generated observable artifacts across both endpoint and network telemetry.

Examples include:

- NTLM-authenticated network logons (Event ID 4624 – Logon Type 3).
- WinRM process creation through `wsmprovhost.exe`.
- Parent-child process relationships associated with remote command execution.
- PowerShell execution policy bypass attempts.
- File creation artifacts for:
  - `PowerView.ps1`
  - `SharpHound.ps1`
- Nmap reconnaissance traffic targeting:
  - RDP (`3389`)
  - WinRM (`5985/5986`)
- Reverse DNS lookup activity (`PTR` requests).
- Incomplete TCP 3-way handshakes (`incomplete SYN`).

Artifacts were analyzed using:
- Suricata IDS alert triage.
- Splunk SIEM queries.
- Sysmon telemetry.
- Wireshark packet captures.

Screenshots and analysis are available in the **telemetry-analysis/** directory.

## 🔎 Investigation & Threat Hunting

Investigation was performed through correlation of:
- Network telemetry
- Authentication logs
- Process creation events
- Sysmon endpoint telemetry

Examples include:

- Identification of NTLM-only authentication activity.
- Correlation of network logons with WinRM process creation.
- Analysis of parent-child process relationships involving `wsmprovhost.exe`.
- Investigation of reconnaissance commands executed remotely.
- Detection of PowerView and SharpHound upload artifacts.
- Wireshark analysis of reconnaissance traffic and WinRM probing activity.

Investigation queries and analysis steps are documented in the **threat-hunting-and-investigation/** directory.

Detection rules were developed to identify suspicious post-compromise enumeration activity within enterprise environments.

These are:

### Splunk Detection Queries

- Detection of suspicious WinRM sessions.
- Correlation of NTLM-only authentication with remote PowerShell execution.
- Detection of offensive tooling uploads.

## 🛡️ Enterprise Mitigations

Recommended mitigations include:

- Restrict WinRM access to approved administrative systems.
- Disable or reduce NTLM authentication where possible.
- Prefer Kerberos authentication within Active Directory environments.
- Enable PowerShell Script Block Logging and Module Logging.
- Deploy Sysmon for improved endpoint telemetry visibility.
- Monitor suspicious use of `wsmprovhost.exe`.
- Restrict upload and execution of offensive tooling.
- Improve east-west network visibility across internal segments.

## 🧑‍💻 Why This Matters

Post-compromise enumeration is often low-noise and blends into legitimate administrative activity.

Attackers frequently abuse:
- Native Windows commands
- Remote administration protocols
- Legitimate PowerShell functionality

As a result, detecting such activity requires:
- Telemetry correlation
- Behavioral analysis
- Visibility across both endpoint and network data sources

rather than relying solely on traditional malware signatures.

---

## 💻 Lab Takeaways

This lab demonstrates several real-world detection challenges:

- Host telemetry provided significantly deeper visibility than packet captures during post-compromise activity.
- Network telemetry alone was insufficient to fully reconstruct attacker behavior.
- In-memory PowerShell tooling creates visibility limitations in default Windows logging configurations.
- Correlating authentication logs, process creation telemetry, IDS alerts, and packet captures was essential to reconstructing the attack chain.
